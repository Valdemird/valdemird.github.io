There's a moment every developer using AI tools has experienced: you ask for a feature, get 200 lines of code, paste it in, and spend the next hour debugging why it doesn't fit your codebase. The code *works* — it just doesn't belong.

The problem isn't the model. It's the process.

After months of building with Claude Code, I developed a workflow that fundamentally changed my results. Instead of treating the LLM as a code generator, I treat it as a **co-architect** — and the difference is night and day.

## The Problem with Prompt-Driven Development

When you open Claude Code and type *"add JWT auth to my API"*, you're asking the model to make dozens of implicit decisions:

- Which JWT library?
- Where do tokens get stored?
- How does it integrate with your existing middleware?
- What's your error handling pattern?
- How are your routes structured?

The model will guess. Sometimes it guesses right. But when it doesn't, you've wasted time on code that needs to be rewritten, not refactored.

<Callout type="important" title="The core insight">
The bottleneck isn't code generation — it's **specification**. If you can clearly define *what* you want, the *how* becomes almost trivial for the model.
</Callout>

## The Spec-Driven Workflow

I built 6 slash commands for Claude Code that enforce a structured pipeline. Each step produces an artifact, and every transition requires human approval. Here's the flow:

```
/spec-create  →  /spec-audit  →  /spec-review  →  /spec-plan  →  /spec-implement
     ↓               ↓               ↓                ↓               ↓
  Raw spec     Enriched spec    Readiness check   Phased plan    Code (phase by phase)
```

Let me walk you through each one with real examples.

## Step 1: `/spec-create` — From idea to structured spec

You give it a name and a rough description. That's it.

<CodeBlock filename=".claude/commands/spec-create.md">
```markdown
# Crear Spec Inicial

Quiero crear un spec para la feature: **$ARGUMENTS**

## Tu tarea

1. Crea el directorio `.claude/specs/` si no existe
2. Toma el input que te di y genera un spec estructurado
3. Guárdalo en `.claude/specs/<nombre-feature>.md`
```
</CodeBlock>

The command takes your vague idea and produces a structured document with:

- **Context**: Why this feature is needed
- **Expected behavior**: Normal cases and edge cases
- **What it should NOT do**: Explicit constraints
- **Acceptance criteria**: Checkboxes you can verify
- **Open questions**: Things *Claude detected* that you need to decide

That last part is the magic. Instead of guessing your JWT library preference, it asks you.

<Callout type="tip" title="Why this works">
By forcing specification before code, you front-load the decisions that cause the most rework. It takes 5 minutes to answer questions in a spec. It takes 2 hours to untangle wrong assumptions in code.
</Callout>

## Step 2: `/spec-audit` — The spec meets reality

This is where it gets interesting. The audit command tells Claude to **explore your actual codebase** and enrich the spec with technical context.

<CodeBlock filename=".claude/commands/spec-audit.md">
```markdown
# Auditar Spec contra el Código

### Paso 2 — Explora el proyecto a fondo

Sin escribir código, explora:
- Estructura general del proyecto
- Patrones de arquitectura usados
- Convenciones de nombrado
- Cómo se manejan los errores actualmente
- Archivos similares a lo que se quiere construir
```
</CodeBlock>

After running this, your spec gets a new section — **Technical Analysis** — with a table of files to modify, files to create, existing patterns to follow, relevant dependencies, and detected risks.

This step prevents the #1 failure mode of AI-generated code: **ignoring existing patterns**. Claude now knows that your project uses a specific error handling approach, a particular folder structure, and certain naming conventions. The code it writes later will *fit*.

## Step 3: `/spec-review` — The readiness traffic light

Before investing time in a plan, this command gives you a clear readiness assessment:

- 🟢 **Ready**: Well-defined, clear enough to implement
- 🟡 **Can proceed with assumptions**: Ambiguous but Claude can make reasonable guesses
- 🔴 **Blocking**: Critical decisions that must be made before any code

This is the checkpoint that prevents wasted effort. If something is red, you fix it now — not after 3 phases of implementation.

## Step 4: `/spec-plan` — Phased implementation plan

The plan command reads the audited spec and generates a step-by-step implementation plan. Each phase is:

- **Small**: 1-3 files touched
- **Verifiable**: Clear "how do I know it's done" criteria
- **Independent**: Can be reviewed before continuing

<CodeBlock filename="Example plan output">
```markdown
### Phase 1 — Database schema and migration
**Objective:** Users table exists with auth fields
**Files to touch:**
- create: prisma/migrations/add-auth-fields.sql
- modify: prisma/schema.prisma
**How to verify:** `npx prisma migrate dev` runs without errors

### Phase 2 — Auth middleware and JWT utilities
**Objective:** Protected routes return 401 for unauthenticated requests
**Files to touch:**
- create: src/middleware/auth.ts
- create: src/utils/jwt.ts
- modify: src/app.ts
**How to verify:** curl unprotected route → 200, protected route → 401
```
</CodeBlock>

<Callout type="warning">
The plan has a hard rule: **maximum 3-4 phases** for medium features. If it needs more, the feature is too big and should be split. This prevents scope creep.
</Callout>

## Step 5: `/spec-implement` — Execute phase by phase

Finally, the implementation command. But notice — it only implements **one phase at a time**.

<CodeBlock filename=".claude/commands/spec-implement.md">
```markdown
# Implementar Fase $2

## Reglas de implementación

- Sigue exactamente los patrones detectados en el análisis técnico
- Si encuentras algo inesperado que cambia el plan,
  **detente y avísame** antes de improvisar
- Usa el estilo de código que ya existe en el proyecto
```
</CodeBlock>

After each phase, Claude:

1. Runs the relevant tests
2. Summarizes what it did and what decisions it made
3. **Asks for your review before continuing**

This is the key difference from "generate all the code at once". If Phase 1 reveals a problem, you can adjust the plan before Phase 2 starts. The cost of change stays low.

## Bonus: Review commands

Beyond specs, I also use two review commands that complement the workflow:

### `/performance`

A structured performance review that checks algorithms, N+1 queries, memory leaks, caching opportunities, bundle sizes, and more. Findings are severity-ranked with concrete fixes.

### `/architecture`

An architecture review covering SOLID principles, coupling, separation of concerns, design patterns, security, and testability. Includes an ASCII diagram of detected layers and a prioritized refactoring roadmap.

## Install these in your project

The setup is simple. Drop the markdown files in your project:

<CodeBlock filename="Project structure">
```
your-project/
├── .claude/
│   ├── commands/
│   │   ├── spec-create.md
│   │   ├── spec-audit.md
│   │   ├── spec-review.md
│   │   ├── spec-plan.md
│   │   ├── spec-implement.md
│   │   ├── performance.md
│   │   └── architecture.md
│   └── specs/
│       └── (generated specs go here)
└── CLAUDE.md
```
</CodeBlock>

Each command is a standalone markdown file with frontmatter that defines its description, allowed tools, and argument format. Claude Code picks them up automatically as `/project:command-name` slash commands.

## The golden rule

> Never skip steps under pressure. The time you "save" by skipping the audit or the review, you pay triple in corrections later.

This workflow isn't about slowing down — it's about **investing 10 minutes in specification to save 2 hours in debugging**. The LLM becomes dramatically more useful when you treat it as a collaborator who needs context, not a magic box that guesses what you want.

Spec-driven beats prompt-driven. Every time.
